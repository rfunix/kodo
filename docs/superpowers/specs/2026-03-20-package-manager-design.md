# Package Manager — Design Spec

**Date**: 2026-03-20
**Status**: Approved
**Scope**: Milestone 6 — `kodo.toml` manifesto, git-only deps, lock file, CLI commands

## Context

Kōdo v0.13.0 has no package manager. Code reuse requires manual file copying. This blocks ecosystem development.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Registry model | Git-only (like Go modules) | Zero infrastructure, agents consume any repo |
| Manifesto format | Minimal TOML | YAGNI — metadados extras adicionados depois |
| Dependency resolution | Flat (one version per dep) | Projetos pequenos, conflitos raros |
| Lock file | `kodo.lock` with commit SHAs | Tags can be moved, SHA is deterministic |
| CLI commands | init, add, remove, update, build | Full lifecycle for agents |

## Manifesto: `kodo.toml`

```toml
module = "my-app"
version = "0.1.0"

[deps]
json = { git = "https://github.com/user/kodo-json", tag = "v1.0.0" }
utils = { path = "../shared-utils" }
```

## Lock file: `kodo.lock`

```toml
[[package]]
name = "json"
git = "https://github.com/user/kodo-json"
tag = "v1.0.0"
commit = "a1b2c3d4e5f6..."

[[package]]
name = "utils"
path = "../shared-utils"
```

Auto-generated by `kodoc add` and `kodoc update`. Should be committed to version control.

## Dependency Resolution

1. Read `kodo.toml` for direct deps
2. For each git dep: clone to `~/.kodo/cache/<sha>/` (or reuse if cached)
3. For each path dep: resolve relative to `kodo.toml`
4. Read transitive deps from each dep's `kodo.toml`
5. If version conflict (same dep at different versions) → error with message
6. Write resolved SHAs to `kodo.lock`

## CLI Commands

### `kodoc init [name]`
Creates project structure:
```
name/
  kodo.toml
  src/
    main.ko
```

### `kodoc add <git-url> --tag <version> [--name <alias>]`
- Adds dep to `kodo.toml`
- Clones repo, resolves SHA
- Updates `kodo.lock`

### `kodoc remove <name>`
- Removes dep from `kodo.toml` and `kodo.lock`

### `kodoc update [name]`
- Re-resolves all deps (or specific dep)
- Updates `kodo.lock` with fresh SHAs

### `kodoc build` (extended)
When `kodo.toml` exists in current directory:
1. Resolve deps from `kodo.lock` (or `kodo.toml` if no lock file)
2. Clone/checkout deps to cache
3. Add dep source paths as available import locations
4. Compile as usual

## Cache

- Location: `~/.kodo/cache/`
- Structure: `~/.kodo/cache/<url-hash>/<commit-sha>/`
- Reused if commit SHA matches
- `kodoc clean-cache` to purge

## Integration with Build

The resolver produces a list of `(name, path)` pairs. The build command adds these paths to the import search path, so `import json { parse }` finds `~/.kodo/cache/.../json/src/lib.ko`.

## New Files

- `crates/kodoc/src/manifest.rs` — parse/write `kodo.toml`
- `crates/kodoc/src/lockfile.rs` — parse/write `kodo.lock`
- `crates/kodoc/src/resolver.rs` — dependency resolution (git clone, path resolve, cache)
- `crates/kodoc/src/commands/init.rs` — `kodoc init` command
- `crates/kodoc/src/commands/deps.rs` — `kodoc add/remove/update` commands
- Modify: `crates/kodoc/src/commands/build.rs` — integrate dep resolution
- Modify: `crates/kodoc/src/main.rs` — wire new subcommands
- New dep: `toml` crate for parsing

## Non-Goals (v1)

- Central registry
- Semver resolution
- Workspace/monorepo support
- Private registry authentication
- Dependency vendoring
